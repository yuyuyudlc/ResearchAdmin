# 科研项目文档管理系统逻辑规约说明书 (System PRD)

## 1. 产品概述

### 1.1 产品定位
本项目是一款专为科研机构、高校实验室及研发团队设计的**科研项目文档管理与协同编辑系统**。系统旨在解决科研协作中资料存储零散、多人编辑版本冲突、敏感数据共享缺乏细粒度控制、协作流缺乏审计追踪等核心痛点，提供安全、高效、透明的一体化协作空间。

### 1.2 核心业务目标
- **文档生命周期集中管理**：提供科研文档的上传、分类、组织、归档、移动及彻底删除的全生命周期流转。
- **多人无冲突协同编辑**：集成 CRDT 协同编辑引擎，支持多人同时在线对富文本内容进行无冲突编辑与段落批注。
- **细粒度多维共享与权限隔离**：支持以工作区成员身份为底座、辅以单文档级 ACL（访问控制列表）的组合校验，兼顾团队协作便利性与科研敏感数据的安全性。
- **版本回溯与合规审计**：全量保存编辑过程中的二进制快照，提供高精度的修改历史追溯；对敏感操作进行完整的安全追踪，满足科研合规审计需求。

## 2. 系统角色与用户模型

在整个系统的生命周期中，用户与权限关系紧密关联。系统定义了四种主要角色：

### 2.1 普通用户 (User)
- **定义**：系统中的基础账号个体。
- **主要行为**：
  - 管理个人账户信息，更新头像、签名及科研头衔（如教授、讲师、学生等）。
  - 在被授权的工作区或特定文档内，进行阅读、评论或协同编辑。
  - 通过全局搜索查找可访问的科研文献。

### 2.2 工作区成员 (Workspace Member)
- **定义**：指在特定项目工作区（Workspace）内拥有显式成员资格的用户。
- **角色细分**：
  - **工作区拥有者 (Owner)**：具有工作区的最高管理权。可修改工作区名称/描述、管理成员名单与角色分配、物理删除工作区等。
  - **工作区成员 (Member)**：工作区内的日常协作人员。具有默认的文档读取与新建权限。

### 2.3 文档拥有者 (Document Owner)
- **定义**：文档的创建者，或者是被指定转移了拥有权的独立用户。
- **主要行为**：
  - 对其拥有的文档配置细粒度共享策略（分配单用户/所有人权限）。
  - 拥有文档的完全控制权（移动、归档、彻底删除及权限收回）。

### 2.4 系统管理员 (System Administrator)
- **定义**：系统的超级管理员，负责全局运维与配置管理。
- **主要行为**：
  - 管理全系统机构树（组织划分、人员归属调整）。
  - 维护用户账号状态（激活、禁用、密码重置）。
  - 审计全系统的安全日志与操作链。

## 3. 核心业务流程与功能规约

```
【文档上传/创建】 ─(自动配置)─> 【初始快照 & 元数据】 ─(配置共享)─> 【工作区继承 / 独立 ACL】
                                                                      │
【审计日志追踪】 <─── 【操作动作 (下载/导出/删除)】 <─── 【协同编辑/在线评论】
```

### 3.1 科研工作区管理 (Workspace)
- **工作区创建**：用户注册后自动创建个人私有空间。用户也可手动创建各类“科研项目工作区”。
- **成员授权**：工作区拥有者可以邀请其他用户加入该空间，并赋予 `member` 或 `owner` 角色。
- **安全约束**：为防止管理真空，任何工作区**必须保留至少一名 Owner**。如果当前用户是该工作区唯一的 Owner，系统应禁止其被移除或退化为普通 Member。

### 3.2 文档分类、组织与移动
- **树状继承结构**：工作区内的文档以树状文件夹/节点结构呈现。
- **文档移动规约**：
  - 文档支持通过配置 `ParentID` 移入不同的父文档节点或直接移至根目录。
  - **防死锁/防断链约束**：在文档树节点进行移动操作时，系统必须在业务逻辑层严密防范**“将自身或父节点移动到自身的子节点下”**。如果发生此类操作，将会导致整条树形链断裂或产生死循环，因此后端必须进行拓扑检测并予以拦截。

### 3.3 文档生命周期流转 (归档与恢复)
- **归档 (Archive)**：
  - 科研文档在项目结题或阶段完成时可执行归档。
  - 归档后的文档在工作区文档树中处于特殊冻结状态（只读，且带“已归档”标签）。
- **恢复 (Restore)**：
  - 归档文档支持被拥有者重新恢复，恢复后自动解除只读状态。
- **彻底删除 (Delete)**：
  - 支持文档级软删除与物理清理，删除后会连带清理关联的 ACL 规则以及正文 Blob 数据。

### 3.4 实时协同编辑与感知
- **多人协同**：基于 CRDT 算法（Yjs），允许多个协作者同时编辑同一份文档，由协同服务负责毫秒级增量分发与冲突合并。
- **状态同步**：系统以本地 Y.Doc 承载内容，协同过程中服务自动监听差异包，在保证协同流畅的前提下自动向主后端刷盘合并快照。
- **用户感知 (Awareness)**：展示协同者的在线头像、实时光标位置及选区高亮。

### 3.5 协作评论与讨论
- **批注插入**：拥有编辑或阅读权限的用户，可在正文特定文字或片段上发起评论（批注）。
- **讨论流**：支持针对特定批注进行多人回复，支持将讨论标记为“已解决”以过滤冗余干扰。

### 3.6 版本历史追溯
- **快照固化**：系统在协同服务触发正式刷盘保存时，自动将最新的 Yjs 二进制状态固化为版本快照。
- **快照对比与还原**：用户可浏览所有的版本历史轴（包含修改人、修改时间），支持对比当前正文与过去某个历史快照的差异（Diff），并支持一键将正文还原到指定的快照状态。

## 4. 文档共享与精细化权限判定逻辑 (ACL)

系统的安全性基于一套**深度融合工作区角色与文档独立 ACL 的多维度权限运算系统**。

### 4.1 权限位图定义 (Bitwise Permissions)
系统使用整型位图（Bitmask）存储和表达权限组合，极大提升后端运算性能：
- `0x01` (1) — **只读 (PermissionRead)**：允许查看文档元数据及正文，允许查看协作评论。
- `0x02` (2) — **编辑 (PermissionEdit)**：允许加入协同编辑、修改正文，允许发表及回复评论。
- `0x04` (4) — **管理 (PermissionManage)**：允许配置分享、修改 ACL 规则，允许归档、转移拥有者。
- `0x08` (8) — **拒绝 (PermissionDeny)**：强行阻断任何访问（最高优先级）。

### 4.2 权限判定核心算法 (Precedence Rules)
当一名用户尝试对文档进行特定操作时，主后端服务必须严格遵循以下**优先级链条**计算其最终权限（优先级由高到低）：

1. **绝对拥有者优先 (Absolute Owner)**：
   - 如果当前用户是**文档的 Owner**，或当前用户是**所属工作区的唯一/联合 Owner**，直接判定为拥有最高管理权限（拥有 `Read | Edit | Manage`，绕过后续 ACL 校验）。
2. **拒绝权限一票否决 (Deny Overrides)**：
   - 检查文档上是否存在针对该用户的 explicit `PermissionDeny` ACL 规则。若有，直接阻断访问。
3. **独立 ACL 覆盖 (Specific ACL)**：
   - 检查该文档上针对该用户的显式 ACL 规则（若有，按此计算并终止）。
   - 检查该文档上是否存在 `public`（所有人公开）的 ACL 规则（若有，按此计算并终止）。
4. **树状层级继承 (Recursive Inheritance)**：
   - 如果文档本身未配置显式 ACL，但其配置了“继承父级”属性，则计算引擎必须沿着 `ParentID` 向上递归寻祖，直至找到配置了显式 ACL 的祖先节点，并应用该 ACL 判定。
5. **工作区身份底座 (Workspace Default)**：
   - 若上述均未匹配，则退回到基本身份校验：如果该用户属于当前工作区的 `member`，则默认拥有该文档的 `PermissionRead | PermissionEdit` 基础权限；否则，判定为不可见。

## 5. 项目已知功能缺失与改进路线

为了真正达成上述 PRD 设计目标，当前项目在产品交互上仍存在以下亟待完善的已知缺失：

### 5.1 协同 WebSocket 通信层缺失
- **现象**：前端目前缺少 `y-websocket` 的 WebApp 通信桥接，实际上并未与 Node.js 协同服务（`:3001`）握手。
- **改进路线**：在前端安装并初始化 `WebsocketProvider`，使其与 `@tiptap/react` 配合，完成真正的实时无冲突协同。

### 5.2 精准用户检索与分享下拉组件缺失
- **现象**：权限分享弹窗中，用户只能手动录入 36 位用户 UUID。普通用户在没有管理员权限的情况下，无法检索并选择想要分享的课题组成员。
- **改进路线**：开发公共用户搜索 API，并在前端引入 `Ant Design` 动态搜索下拉选择框。

### 5.3 协同感知与批注讨论的 UI 整合
- **现象**：当前尚未提供是在线用户光标展示以及段落评论的交互面。
- **改进路线**：在编辑器侧边栏构建常驻的“协作者状态”与“评论批注抽屉”。
