# 科研项目文档管理系统后端架构设计

## 1. 文档目标

本文档用于定义科研项目文档管理系统后端的模块边界、核心链路、权限原则和存储职责，作为后续数据库设计、接口设计和服务实现的基础。

当前技术前提如下：

- Web 框架：`Gin`
- 语言：`Go`
- ORM：`GORM`
- 鉴权：`JWT`
- 协同编辑：`Tiptap + Yjs`
- 前端：`React + Ant Design`

本文档优先解决“后端负责什么、边界如何划分、核心规则是什么”，不展开到具体代码实现。

## 2. 设计目标

后端设计目标如下：

- 以“文档”为中心组织业务，统一承载上传、编辑、共享、检索、审计与历史能力。
- 所有敏感能力由后端完成权限校验，前端只负责界面呈现和交互引导。
- 将“文件存储”“元数据管理”“协同状态”“审计记录”拆开，避免后期耦合失控。
- 兼容两类文档形态：普通文件型文档和富文本协同型文档。
- 支持后续扩展对象存储、全文检索服务、消息通知和备份恢复能力。

## 3. 后端职责范围

后端负责以下能力：

- 用户认证与身份识别
- 用户组和成员关系管理
- 文档元数据管理
- 文件上传、下载、预览地址签发
- 文档共享与权限计算
- 历史快照登记与恢复
- 评论、回复与处理状态
- 协同编辑会话鉴权与状态管理
- 文档检索和权限过滤
- 审计日志记录与查询
- 备份与恢复任务管理

后端不负责以下内容：

- 富文本编辑器的具体渲染与交互
- Yjs 本地冲突合并算法本身
- 前端页面级权限展示逻辑

## 4. 总体分层

建议后端采用典型分层结构：

```text
HTTP Handler
-> Application Service
-> Domain Logic
-> Repository
-> DB / Object Storage / Search / Yjs Store
```

各层职责如下：

- `handler`：接收 HTTP 请求、参数校验、响应转换，不直接写业务规则。
- `service`：编排业务流程，例如上传文档、分配权限、恢复版本。
- `domain`：承载核心业务规则，例如最终权限计算、历史快照生成条件。
- `repository`：封装数据库读写与外部存储访问。
- `infrastructure`：数据库、对象存储、搜索引擎、Yjs 状态存储、日志与配置。

## 5. 模块划分

### 5.1 Auth 模块

职责：

- 用户登录
- JWT 签发与校验
- 当前用户身份解析
- 协同会话访问票据签发

核心规则：

- 所有业务请求都基于用户身份处理，不允许匿名访问敏感资源。
- 协同编辑连接必须重新校验权限，不能仅依赖页面已登录状态。

### 5.2 User / Group 模块

职责：

- 用户基本信息管理
- 用户组管理
- 用户组成员关系维护

核心规则：

- 用户组是批量授权的载体，不直接等于业务角色。
- 用户角色用于系统级权限，用户组用于文档共享范围。

### 5.3 Document 模块

职责：

- 文档创建、上传、更新、归档
- 文档元数据维护
- 文件型文档与协同型文档统一抽象
- 预览、下载能力的访问控制

核心规则：

- 文档是系统中心对象。
- 文件本体与元数据分开存储。
- 文档创建时必须明确拥有者。

### 5.4 Permission 模块

职责：

- 管理 `doc_acl` 授权记录
- 计算最终访问权限
- 支持继承授权和单文档覆盖

核心规则：

- 后端统一计算最终权限，前端不参与权限判定。
- 权限变更必须即时生效。
- 检索、预览、下载、评论、编辑都依赖最终权限位掩码。

### 5.5 History 模块

职责：

- 记录文档历史快照
- 维护历史操作说明
- 支持版本查看与恢复

核心规则：

- 历史记录记录“正式状态”，不记录每次临时输入。
- 恢复旧快照会生成一条新的历史记录，而不是覆盖旧记录。

### 5.6 Collaboration 模块

职责：

- 为 Yjs/Tiptap 协同编辑提供会话鉴权
- 管理在线协作者状态
- 负责协同状态与历史快照的衔接

核心规则：

- 协同编辑与历史记录是两条链路。
- Yjs 实时状态负责同步，正式保存后才沉淀为历史快照。
- 只有具备编辑权限的用户可以加入编辑会话。

### 5.7 Comment 模块

职责：

- 文档评论、回复、状态变更
- 片段级批注关联

核心规则：

- 评论能力至少要求可见权限。
- 修改正文的能力与评论能力分离，避免“只读但可评论”的场景无法表达。

### 5.8 Search 模块

职责：

- 元数据检索
- 条件筛选
- 全文搜索
- 搜索结果权限过滤

核心规则：

- 搜索结果必须先经过权限过滤，再返回给前端。
- 未授权文档不应出现在检索结果中。

### 5.9 Audit 模块

职责：

- 记录关键操作
- 支持按用户、文档、时间范围查询

核心规则：

- 审计日志不可被普通用户篡改。
- 敏感动作和关键失败动作都应纳入审计范围。

### 5.10 Backup 模块

职责：

- 记录备份任务
- 记录恢复任务
- 展示备份状态

核心规则：

- 文档文件和业务元数据必须同时纳入备份策略。
- 恢复任务本身也应被审计。

## 6. 文档类型设计

系统中建议统一使用 `documents` 作为主实体，但区分两种内容承载方式：

- `file`：普通附件型文档，例如 PDF、Word、Excel、图片、音频、实验数据文件。
- `rich_text`：在线协同型文档，正文由 Tiptap/Yjs 管理。

统一抽象的原因如下：

- 前端文档列表、权限、评论、审计、历史等能力都围绕“文档”统一工作。
- 不同内容类型只影响内容存储方式，不影响共享、检索和访问控制模型。

## 7. 核心业务链路

### 7.1 上传链路

```text
创建文档
-> 校验创建权限
-> 上传文件或初始化富文本文档
-> 保存元数据
-> 建立拥有者关系
-> 生成初始历史记录
-> 写入审计日志
```

### 7.2 共享链路

```text
拥有者提交共享配置
-> 校验拥有者身份
-> 写入用户授权 / 组授权
-> 重新计算访问策略
-> 写入审计日志
```

### 7.3 协同编辑链路

```text
用户请求协同会话
-> 校验可编辑权限
-> 签发协同连接凭证
-> Yjs 同步实时状态
-> 用户正式保存
-> 生成历史快照
-> 写入审计日志
```

### 7.4 检索链路

```text
用户输入检索条件
-> 查询元数据 / 全文索引
-> 按最终权限过滤结果
-> 返回文档列表
```

## 8. 权限模型原则

建议采用单张 ACL 授权表 `doc_acl`，并使用整型位掩码（bitmask）表示权限，不使用多布尔字段，也不使用权限字符串枚举。

建议常量定义如下：

- `READ = 1 << 0`（二进制 `0001`，十进制 `1`）
- `EDIT = 1 << 1`（二进制 `0010`，十进制 `2`）
- `MANAGE = 1 << 2`（二进制 `0100`，十进制 `4`）
- `DENY = 1 << 3`（二进制 `1000`，十进制 `8`）

常见组合：

- 可读：`1`
- 可编辑（读 + 写）：`READ | EDIT = 3`
- 管理者（读 + 写 + 管理）：`READ | EDIT | MANAGE = 7`
- 显式不可见（强制阻断）：`DENY = 8`

建议的权限计算规则如下：

1. 文档拥有者拥有最高权限。
2. 单用户授权优先于用户组授权。
3. 单文档授权优先于继承授权。
4. 未命中任何授权时默认拒绝。
5. 搜索、详情、下载、协同会话都必须复用同一套权限计算逻辑。
6. 系统管理员由系统角色体系单独判定，不直接写入文档 ACL 行。
7. `DENY` 位优先级最高，只要命中 `DENY`，最终结果必须直接拒绝。

不可见需要区分两类：

- 隐式不可见：无 ACL 记录且无继承权限，最终权限为 `0`。
- 显式不可见：ACL 显式存在 `DENY`（`8`），强制拒绝。

建议决议伪代码：

```text
merged = 0

for each matched acl in doc_acl:
		merged = merged | acl.permission_bit

if (merged & DENY) != 0:
		return 403 Forbidden

return merged
```

前端消费约定：

- 后端返回当前用户最终权限整数 `user_perm`，不下发复杂 ACL 规则。
- 前端用按位与判断按钮态，例如：
	- `canEdit = (user_perm & EDIT) == EDIT`
	- `canManage = (user_perm & MANAGE) == MANAGE`

## 9. 历史与协同关系

必须明确区分两个概念：

- `实时协同状态`：由 Yjs 驱动，适合承载临时编辑过程。
- `正式历史快照`：由业务保存动作触发，适合审计、追溯和恢复。

对应规则如下：

- 不将每次 Yjs update 直接视为业务版本。
- 只有正式保存、上传新文件、恢复历史、导入新内容等业务动作才生成历史快照。
- 恢复历史快照后，当前文档内容更新，并追加一条新的历史记录。

## 10. 存储设计建议

建议至少拆分以下存储职责：

- 关系型数据库：用户、文档元数据、权限、评论、历史索引、审计日志
- 对象存储或本地文件存储：附件文件、预览转换产物、历史文件快照
- Yjs 状态存储：协同编辑增量状态或压缩快照
- 搜索索引：全文内容和可筛选字段

不建议将大文件正文、全文检索索引和业务主表全部混放在一张数据库表中。

## 11. 非功能设计要求

### 11.1 安全性

- 所有接口必须要求身份认证。
- 所有敏感操作必须在后端重复校验权限。
- 下载、共享变更、版本恢复等行为必须写审计日志。
- 协同连接必须校验连接身份和文档权限。

### 11.2 可维护性

- 模块边界清晰，不在 handler 中直接写复杂业务逻辑。
- 权限计算抽为统一服务，避免每个模块自行判断。
- 历史记录和审计日志的写入逻辑统一封装，避免遗漏。

### 11.3 可扩展性

- 支持未来接入对象存储。
- 支持未来接入搜索引擎，例如 Elasticsearch 或 OpenSearch。
- 支持后续增加更多角色策略和共享模板。
